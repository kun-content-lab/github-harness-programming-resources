# 本期视频资源地图

本仓库只整理 GitHub 和 Harness Programming 重叠的内容。

## 资源边界

| 纳入 | 不纳入 |
|---|---|
| GitHub 如何承接需求、任务、证据、验收 | 记忆系统、内容选题系统、视频制作系统 |
| Discussion / issue / PR / comment / board 的工作流 | 命令行教程或工具背诵 |
| 最小三件套模板 | 业务材料、账号配置、个人路径 |
| AI 资料索引站示例 | 和本期主题无关的专项工程技能 |

## 视频里要讲清楚的核心

1. 聊天窗口适合对话，但不适合承接长项目。
2. GitHub 的双向链接、issue、comment、PR、board，可以把 AI 工作变成可追踪系统。
3. Discussion 先负责需求确认，不要直接跳到执行。
4. Issue 承接任务，Evidence comment 承接完成证据。
5. 这三件事连起来，就是一个最小 Harness Engine。

## 观众可以拿走什么

| 资源 | 观众怎么用 |
|---|---|
| 需求 Discussion 模板 | 先把想做什么、给谁做、第一版做到哪里聊清楚 |
| 任务 issue 模板 | 把确认后的需求拆成一张可执行任务卡 |
| 证据 comment 模板 | 约束 AI 完成后交代证据和下一步 |
| AI 资料索引站示例 | 看一个非代码项目如何使用这套方法 |
| Mermaid 图 | 放进自己的 README、PPT 或项目说明 |

## 不要误解成什么

这套方法不是要求所有人都学会 GitHub 的全部功能。

你真正要掌握的是：把 AI 放进一个有需求、有任务、有证据、有验收的系统里。今天用 GitHub，明天换成别的工具，只要这条 loop 还在，方法就还成立。
